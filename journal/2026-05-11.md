# Journal — May 11, 2026

## What We Did

Dug into the WindowConfigurator codebase for the first time in a while. The goal was simply to understand where things stood — what was here, what wasn't, and whether this was a viable starting point for something real.

## What We Discovered

The project is a backup from an earlier phase of development — specifically from when the system was being pivoted away from a NALP commerce integration toward a CRM-agnostic model. It's not as complete as I had hoped going in, but what's here is actually the hard part.

**The configurator engine is mature and works.** The fractional inch measurement system, the SVG live preview, the grille pattern geometry, the 2D pricing interpolation — all of that is real, battle-tested code that was originally built for a production commerce system. That's the part that takes the most domain expertise and the most time to build. It's done.

What's missing is essentially the shell around it — persistence, auth, the external API layer. That's all well-understood, well-solved territory in .NET. It's plumbing. It's buildable.

---

## The Architecture Clarity

The biggest thing that came out of today was getting clear on the architecture — specifically the CRM integration model.

The original instinct was right: **the CRM calls us, we don't call the CRM.** This keeps the system completely CRM-agnostic. No Salesforce adapter, no HubSpot adapter, no maintenance burden when a client switches CRMs. The CRM expert on the client's side owns their integration. We just expose a clean API and fire webhooks when things happen.

We also worked through the prospect-initiated flow — where someone lands on a dealer's website, clicks "Build My Window," gets a magic link to their email, configures their window, and the result gets pushed to the dealer's CRM as a new lead. The key insight was that this flow can stay unidirectional too. Instead of us calling the CRM to create the lead, we fire a generic webhook to a URL the dealer registered. Their CRM workflow handles it. We never need to know what CRM they're using.

That keeps the whole system clean. Two flows, same pattern, no adapters anywhere.

---

## The Two Flows

**Flow 1 — CRM-Initiated (B2B/Contractor)**
CRM authenticates → creates a session → contractor configures in our UI → result webhooks back to the CRM.

**Flow 2 — Website-Initiated (B2C/Prospect)**
Prospect clicks a link → enters email → gets a magic link → configures their window → result webhooks to the dealer's registered callback URL.

## What's Next

The roadmap is clear:

1. **Persistence** — swap the in-memory repository for a real database (EF Core + PostgreSQL most likely)
2. **External API layer** — versioned REST endpoints for CRM session handoff (`/api/v1/sessions`)
3. **Auth** — API key or OAuth client credentials for B2B, magic link tokens for B2C
4. **Webhook dispatch** — fire-and-forget with retry logic when a session completes
5. **Multi-tenancy** — per-client config for callback URLs, branding, product line access
6. **Configurator modernization** — fix the stubbed endpoints, wire up the save flow, eventually modernize the frontend
