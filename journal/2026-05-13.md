# Journal — May 13, 2026

## What We Did

Spent the day doing two kinds of work on WindowConfigurator: getting the development environment back into a sane state in VS Code, and then using that clearer footing to think through what this project actually needs to become a real product.

The theme of the day was separating what the project *appears* to be from what it *actually is*.

---

## Debugging And Environment Work

The first issue was practical: the app was not launching correctly in VS Code.

That turned out not to be a .NET or build problem at all. The web app itself ran fine. The real issue was the debug configuration. VS Code was set up to launch Chrome against the wrong port, and once that was fixed it was still opening the app at the bare root URL, which currently returns a 404 because the default route expects an `id` and maps straight into `OrderItem/Index`.

So the environment work today was mostly about making the tooling reflect the way the app actually behaves.

### What got fixed

- Replaced the broken browser-only launch configuration with a real ASP.NET Core debug profile in `.vscode/launch.json`
- Added a proper `.vscode/tasks.json` build task so VS Code has a working prelaunch build step
- Corrected the project path used by the build task
- Updated the debugger launch behavior so it opens a known demo route instead of the root path that currently 404s
- Confirmed the app builds and runs cleanly under the installed Windows .NET 10 SDK
- Confirmed that WSL on this machine does not currently have `dotnet` installed, so WSL is not yet a viable runtime path for this repo

In other words: the project is runnable in VS Code now, but the exercise exposed something important about the app itself — it is still highly demo-shaped and route-fragile.

## What We Discovered About The Codebase

Once the app was running, the architectural review got sharper.

The main realization was that this version of the project has very little true server-side application logic. It is an ASP.NET Core shell around a mature configurator engine that still mostly lives in the browser.

The server is currently doing a few important but thin things:

- serving MVC views and static assets
- returning JSON template and pricing files
- holding some in-memory sample repository data
- wiring routes and DI

But it is **not** yet doing the high-trust work that matters for a real product:

- no real persistence
- no real save or complete flow
- no server-side authoritative pricing
- no server-side authoritative validation
- no real session model
- no external API layer yet

That matters, because the current architecture means the browser is still doing things that should eventually belong to the server.

---

## Pricing Reality Check

One specific question came up that was worth answering directly: whether pricing had already been implemented server-side so that a user could not simply open DevTools and manipulate the result.

The answer, in this repo as it stands today, is no.

The pricing data is served by the backend, but the actual price calculation is being performed in JavaScript in the browser. The final displayed amount is therefore not server-authoritative in the current implementation.

That was a useful clarification because it sharpened the roadmap immediately. A trustworthy version of this system needs server-side price recomputation before any meaningful completion or submission occurs.

## Architecture Clarity That Came Out Of Today

The most valuable work of the day was not code. It was boundary setting.

A lot of ambiguity got resolved by recognizing that WindowConfigurator probably should **not** be trying to become a generic order management platform.

The better framing is:

- the CRM owns the downstream sales lifecycle
- WindowConfigurator owns pre-submission configuration state
- the server should become authoritative for sessions, catalog resolution, validation, pricing, and completion

That led to a more practical mental model for the system.

### Product line handling

A key insight was that product line usually should not be a casual, per-window UI choice. In real jobs, product line is often chosen at the order or project level. If the CRM owns the order, then the CRM or quote/session context should usually provide the product line into the configurator.

So product line becomes session input, not arbitrary in-session user-owned state.

### Website flow complication

Another important realization: the prospect flow creates pressure above the single-window level.

If a future customer lands on a dealer's site and wants to configure and price *several* windows, then a single configurator session is not enough. That flow needs a container for multiple configured items and an aggregate total.

That led to the conclusion that the real model probably needs three levels:

1. `Tenant`
2. `QuoteSession` or `ProjectSession`
3. `ConfiguredWindowItem`

That keeps WindowConfigurator useful without forcing it to become every client's full order system.

The cleanest wording that came out of today was this:

**WindowConfigurator should own pre-order configuration aggregation. The CRM should own post-submission sales lifecycle.**

That feels right.

## Roadmap Work

After working through the architecture questions, the next step was to turn the discussion into something persistent and actionable.

A phased implementation roadmap was written and saved to the repo as:

`implementation-roadmap.md`

The roadmap is ordered by risk reduction and effort, not by conceptual neatness. That was deliberate.

### The sequence we settled on

1. Stabilize the demo surface and remove misleading UI affordances
2. Define the real backend shape with session-oriented models
3. Introduce persistence for sessions and configured items
4. Move catalog resolution behind the server
5. Port authoritative pricing to the server
6. Add server-side validation at completion
7. Add a minimal session-based API surface
8. Add CRM handoff and tenant integration features
9. Treat any UI port or modernization as optional and later

This was the biggest concrete deliverable of the day. It converts a bunch of architectural pondering into an actual execution path.

## What Changed In My Understanding

Before today, it was easy to think of the missing work as mostly “backend plumbing.”

After today, the picture is more precise:

- the hard configurator behavior already exists
- the server shell exists, but it is not yet authoritative
- the product boundary has to be corrected before implementation accelerates
- the right next move is not redesigning the UI
- the right next move is putting a real session, pricing, validation, and persistence core behind the existing configurator

That is a much stronger place to be than vague ambition.

## What's Next

The next logical step is to turn Phase 1 of the roadmap into concrete models and contracts:

1. define `Tenant`
2. define `QuoteSession`
3. define `ConfiguredWindowItem`
4. define the minimum API contract for creating, loading, updating, and completing sessions

That work should be small enough to begin immediately, and it will set up everything that follows.

## Reflection

Today was one of those useful engineering days where fixing a small practical problem opened the door to better architectural thinking.

The debugging issue in VS Code forced a close look at how the app really launches and what the app actually assumes. That, in turn, exposed how much of this project is still a server-hosted client app rather than a true service-backed product.

That is not bad news.

It actually makes the path forward clearer: keep the UI, keep the configurator engine, stop pretending the order shell is real, and build the authoritative backend core that the product actually needs.
