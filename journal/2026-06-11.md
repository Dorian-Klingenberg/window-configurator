## Skills Analysis — Interview-Me and Idea-Refine

Twenty-four skills from [addyosmani/agent-skills](https://github.com/addyosmani/agent-skills) were analyzed for fit. The initial framing was wrong twice and worth recording.

**First wrong framing:** "adopt now" was interpreted as "import directly." Dorian corrected: skills earn a sandbox, not a direct slot. The sandbox is where the skill proves its value (or fails to).

**Second wrong framing:** the requirements-gathering skills (`interview-me` and `idea-refine`) were initially rated low priority as "not directly applicable to implementation." Dorian corrected: they are the *highest* priority because requirements gathering is precisely where difficulty lives. The question isn't "how do we build Slice D?" — it's "what does Slice D actually need to do?"

This is worth remembering for future skill evaluations. When assessing a skill, ask "where in the workflow does the user actually struggle?" not "how similar is this to what we already do?"

**Sandbox 001 — `interview-me`:** Five-step structured self-interview (Hypothesize → One question → "Should want" signals → Restate → Confirm). Hypothesis: the output structure (Outcome / User / Why now / Success criteria / Binding constraint / Out of scope) maps directly to REQ-D-NNN format, making the translation step mechanical rather than creative. Test case: Slice D "demo UX hardening" scope.

Hypothesis: Phase 3's "Not Doing" list — explicit scope exclusions with reasoning — is the missing artifact from the current slice requirements format. Without it, requirements drift because there is no written boundary. Test cases: Slice D or the Agile V adoption initiative itself (which is fuzzier and would stress-test the skill's convergent phase).

## Agile V Paper — Case Study Analysis

The HIL (Hardware-in-the-Loop) case study in the paper tested the Agile V Infinity Loop against a ~500 LOC Python project with 8 requirements and 54 tests across two cycles. Key empirical results:

- **6 human prompts per cycle** — the H3 benchmark. This is measurable and meaningful as a human interaction cost.
- **6.75:1 test-to-requirement ratio** (54 tests / 8 requirements).
- **10 Red Team findings in Cycle 2** (6 MAJOR, 4 MINOR) — all resolved before merge.
- **Cycle 2 is the crucial one**: the system passed Cycle 1 with 100%. It was only in Cycle 2 (after a requirements change request) that the Red Team found significant gaps. This demonstrates the value of the independent verification pass — it catches what implementation-aligned tests miss.

Three things were added to the project immediately based on this:

1. **Three new Tier 2 metrics** in `metrics/baseline.md`: human prompt count per cycle (benchmark: 6), test-to-requirement ratio (benchmark: 6.75:1), Red Team findings per cycle (benchmark: 10 in Cycle 2).

2. **ATM added to Documentation Conventions in `AGENTS.md`**: `requirements/atm-slice-d.md` is now a required closing artifact. Format: REQ-X-NNN | Implementation (file:class/method) | Test (file:method). One row per requirement.

3. **Red Team Protocol added to Testing Conventions in `AGENTS.md`**: the agent or prompt session designing tests must receive only the requirements file — never the implementation. This is a hard rule. The paper's central finding is that the structural separation between Build Agent and Test Designer (no shared context) is what makes H2 (100% requirement-level verification) achievable. Without it, you have an author reviewing their own work.

**What was not adopted from the paper:** the `.agile-v/` state directory (our journals + ADRs + lessons + metrics serve the same purpose without a parallel structure), the Logic Gatekeeper role (no hardware constraints), and the GMP/GxP compliance mapping (no regulatory context).

## WSL Environment Diagnosis

During the session it was discovered that several commands had been running through PowerShell or Git Bash (MSYS2) instead of WSL. The Bash tool in Claude Code runs inside Git Bash (MSYS2), not WSL. This means `/mnt/d/...` paths get mangled to `C:/Program Files/Git/mnt/d/...` unless path conversion is disabled.

The fix: `MSYS_NO_PATHCONV=1 /c/WINDOWS/system32/wsl -- <command with /mnt/d/... paths>`. This was saved to memory (`feedback-wsl-msys-pathconv.md`).

For commands that stay in Git Bash (no WSL needed), use `/d/...` paths instead of `/mnt/d/...`.

## On Whether a Lesson Is Due

The existing lesson `lesson_2026-06-11_agile-v-experiment-design.md` covers the strategic design well. The second session was more operational — executing the plan rather than designing it.

One insight from this session is genuinely lesson-worthy but hasn't been written up as a formal lesson yet: **the requirements-gathering skill inversion**. When evaluating new tools, the correct question is "where does the user actually struggle?" not "is this tool similar to what we already do?" Skills that address the hard part earn a sandbox. Skills that automate the easy part are lower priority regardless of their apparent relevance.

Whether that becomes a formal lesson is deferred to after Sandboxes 001 and 002 close — at that point there will be empirical data on whether `interview-me` and `idea-refine` actually solve the problem, which would make the lesson more grounded.
