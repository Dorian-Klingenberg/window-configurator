## Phase 2: Session Persistence

The goal was to replace the in-memory repository placeholder with a real EF Core persistence layer backed by SQLite for development.

### What was built

- **EF Core packages** — `Microsoft.EntityFrameworkCore.Sqlite` and related packages added to the `.Data` and `.Tests` projects
- **`WindowConfiguratorDbContext`** — DbContext with `DbSet<TenantEntity>`, `DbSet<QuoteSessionEntity>`, `DbSet<ConfiguredWindowItemEntity>`; relationship mappings via fluent API
- **`EfQuoteSessionRepository`** — full implementation of `IQuoteSessionRepository` using async EF Core queries
- **`IQuoteSessionRepository`** interface — `CreateAsync`, `GetByIdAsync`, `GetByTenantAsync`, `AddItemAsync`, `UpdateItemAsync`
- **Integration tests** — `Repositories/QuoteSessionRepositoryTests.cs` with SQLite in-memory, verifying: create, retrieve, add item, update item, status filter, cross-tenant isolation
- **`Program.cs`** — wired up DbContext (SQLite dev path at `app.db` in project root) and registered repository

### Key decisions

**SQLite for development, proper EF Core for everything else.** The DbContext is provider-agnostic; swapping to PostgreSQL or SQL Server for production is a one-line change in `Program.cs`.

**Tests use `SqliteConnection("Data Source=:memory:")` + `EnsureCreated()`**, not the EF InMemory provider. This was a deliberate choice documented in `AGENTS.md` — the InMemory provider doesn't run real SQL and misses constraint and query bugs.

**`app.db` lives in `WindowConfigurator/WindowConfigurator/`** (project root alongside `appsettings.json`). On first run, EF Core creates it automatically via `EnsureCreated()` in `Program.cs`.

## Phase 3: Catalog Resolution

The goal was to stop the controller from always returning the EnergySaver template regardless of which product line the session was configured for.

### What was built

- **`ProductLineCatalog`** — new model in `WindowConfigurator.Data/Catalog/` capturing key, name, manufacturer, item template filename, and section template filename
- **`ICatalogService`** — interface with `GetProductLine(key)`, `ListAll()`, `GetAllowedForTenant(tenant)`
- **`CatalogService`** — singleton implementation registering the three product lines (EnergySaver 2500, Apex, Carriage) and resolving by key
- **`ITemplateReader`** — interface extracted from `WindowConfiguratorDataHelper` to allow test doubles in controller tests; production implementation delegates to the existing file-serving logic
- **`OrderItemController` refactored** — constructor now takes `IQuoteSessionRepository`, `ICatalogService`, `ITemplateReader`; `GetOverview` and `SectionTemplate` resolve the session, look up the product line key, ask the catalog for the right filename; returns 404 on unknown session
- **Controller tests** — `Controllers/OrderItemControllerTests.cs` with 6 tests: Apex session → Apex template, EnergySaver session → EnergySaver template, unknown session → 404
- **`Program.cs`** — registered `ICatalogService` as singleton, registered `ITemplateReader`

### Key decisions

**`ITemplateReader` is the testability seam.** The existing `WindowConfiguratorDataHelper` reads files from the filesystem and depends on `IWebHostEnvironment`. Rather than threading that into controller tests, the interface was extracted so tests can inject a stub that returns predictable content. The production wiring remains unchanged.

**`ICatalogService` registered as singleton** — the catalog is static and read-only; there is no reason to recreate it per request.

## Cleanup: Removing Premature Save Endpoint

**Problem discovered:** GPT-mini had implemented both `Save` and `Complete` endpoints. This caused confusion:
- Save was meant for draft state (multi-window sessions in progress)
- Complete was meant for finalization (server pricing + webhook trigger)
- But the multi-window UI doesn't exist yet (that's Phase 9)
- The item resolution logic (`ResolveDraftItem`) had a fallback to `session.Items.FirstOrDefault()` which meant all users without explicit item IDs would edit the same first item
- This masked the fact that multi-item support wasn't actually working

**Session collision risk:** The `ResolveIndexSessionAsync` fallback to `GetFirstAsync()` meant multiple users launching the app without a session ID would all land on the same session. This is fine for local dev but would be a disaster in production.

**Decision:** Remove `Save` entirely for now. Phase 4 is single-item sessions. Save only makes sense once the UI supports adding/switching between items (Phase 9). Keeping it now adds complexity with no user benefit.

### What was removed

- **`OrderItemController.Save`** endpoint and `Post` alias
- **`ClearCompletionFields`** helper method (no longer needed without Save)
- **Save button** from `Views/OrderItem/Index.cshtml`
- **Save button click handler** from JavaScript
- **`Save_WithExistingSession_UpdatesTheDraftItem` test** (no longer valid)

### What was added

- **`OrderItemController.CreateDevSession`** endpoint — creates a fresh session with a new Guid and redirects to the configurator
- **Route `/new`** mapped to `CreateDevSession` for easy development access
- Each visit to `/new` creates a unique session, eliminating the multi-user collision problem for dev testing

### Behavior now

- User visits `/new` → server creates session with unique ID → redirects to `/OrderItem/Index/{guid}`
- User configures window → clicks "Submit Window"
- Server computes authoritative price, marks item `Completed`, marks session `Completed`
- Success alert shows server's authoritative price

**The `GetFirstAsync()` fallback stays for now** because it's useful during development to hit `/` and land on the demo seed session. In production, every entry point (CRM webhook, email magic link) will pass an explicit session ID.

- **Build:** Clean, no warnings
- **Tests:** 76 passing (down from 77 after removing the Save test)
- **User flow:** Visit `/new` → configure → submit → see authoritative price

## Pricing Port Follow-Up

Later the same day, the pricing cross-validation work was revisited because one failing test still claimed a JavaScript result of `$1812.33` while the C# service returned `$544.10`.

The important discovery was that the failing expectation was based on a stale screenshot narrative, not on executing the exact payload through the legacy browser engine. Running `pricing.js` directly against the same payload and `priceInfo.json` data produced **`$1066.75`**.

Two concrete mismatches were then fixed in the C# port:

- **Frame vs. outside measurements** — the browser prices frame color from `osmWidth` / `osmHeight`, but brickmould from frame `width` / `height`. The C# input model was only carrying one pair.
- **Legacy brickmould interpolation quirk** — when width exceeds the last breakpoint, the JavaScript brickmould calculator does not clamp cleanly; its zero-width branch reuses the breakpoint PPI as a slope, so the returned `upperHeightPpi` keeps growing. The C# port had been using a mathematically cleaner clamped interpolation instead.

The pricing service now matches the legacy JavaScript behavior for that case, and the completion path now builds pricing input from both frame and outside measurements.

### Verification

- `dotnet test .\\WindowConfigurator.sln --nologo` -> 140 passing
- Cross-validation case `LargeWindow_TwoSections_2InchBrickmould_MatchesJavaScript` now passes at **`$1066.75`**

## Pricing Matrix Audit

The next pass broadened the pricing comparison work from one disputed case to a generated matrix of product-line, section-count, grille, SDL, brickmould, pane, crank, and size combinations.

### What was added

- **Generated fixture** — `WindowConfigurator.Tests/Pricing/generate-pricing-comparison-fixture.js` now executes the legacy browser pricing engine directly and emits a large comparison fixture
- **Matrix regression tests** — `WindowConfigurator.Tests/Pricing/PricingComparisonMatrixTests.cs` now checks that:
  - all genuinely in-range cases match the legacy JavaScript exactly
  - past-final-breakpoint cases still expose parity gaps instead of silently disappearing

### What was learned

The first matrix failures were not C# pricing bugs. They were fixture-construction bugs:

- some single-section cases omitted pane configuration from the comparison payload, so the server side was pricing them without the pane surcharge
- some multi-section cases assigned different pane configurations per section even though the legacy JavaScript prices pane configuration from the window-level selection
- some cases were marked "in range" based only on the style grid even though the selected grille, SDL, pane, or brickmould grid had already gone past its supported breakpoint range

Once the fixture was corrected, the actual pricing story became much clearer:

- **49,460 in-range cases** matched exactly
- **31,450 past-breakpoint cases** were generated
- **21,046** of those past-breakpoint cases diverged, confirming that the remaining parity gap is specifically the legacy extrapolation behavior once width and/or height go beyond the final supported breakpoints

### Representative overflow examples

- `EnergySaver 2500|single|Casement|White|None|Dual|None|None|width-overflow` — JS **`$2442.96`**, C# **`$569.05`**
- `EnergySaver 2500|single|Casement|White|None|Dual|None|None|width-height-overflow` — JS **`$7003.05`**, C# **`$624.30`**
- Largest observed delta in the generated matrix: `EnergySaver 2500|multi-3|Fixed Sash+Fixed Sash+Fixed Sash|White|1 1/2 Inch - Fin|p9` at **`$211,196.18`**
