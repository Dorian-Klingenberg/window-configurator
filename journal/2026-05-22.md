## VS Code Debug Reliability Update (Temporary)

Added a branch-local VS Code prelaunch task:
- `reset-dev-db-windowconfigurator-temporary`

What it does:
- deletes `windowconfigurator.db` (+ `-shm` / `-wal` when present) before configurator build/debug startup.

Why this decision was made:
- local debug runs were repeatedly failing with stale-schema SQLite errors (`no such column: t.ApiKeyExpiresAtUtc`) after model evolution across branches.
- for current Slice A velocity, deterministic startup is more valuable than preserving local dev data.

Constraint:
- this is a temporary development workaround, not a substitute for long-term migration discipline.

Follow-up debug fix:
- VS Code launch profiles were corrected to use `%s` in `serverReadyAction.uriFormat` (required by VS Code), and both workspace-level `.vscode` folders were aligned so run profiles behave consistently regardless of which root folder is open.

## Ownership Clarification: Multi-Window Flow

Decision:
- the ability to create/add another window after completing one belongs to the **configurator UX**, not the mock contractor website host shell.

Why:
- backend session/item APIs already support multiple windows in a quote session.
- host-site scope is launch/orchestration; putting core quote-item lifecycle UX there would duplicate configurator responsibilities and create drift.
- keeping multi-window controls in the configurator preserves a single source of truth for quote-building behavior.

Plan impact:
- keep Slice A focused on landing + launch.
- queue multi-window in-session UX for the next configurator-facing slice (likely Slice C/D branch scope).
